**Цель:** Переносить знания и выполнять задачи между сессиями.

## Cron Jobs

### Когда использовать:
- Регулярные задачи (ежедневно, еженедельно)
- Напоминания
- Автоматические исследования

### Два режима:

**1. Main session (`sessionTarget: "main"`):**
- SystemEvent попадает в очередь heartbeat
- НЕ прерывает активную сессию
- Подходит для: напоминания, проверки статуса
- `wakeMode: "now"` — триггерит heartbeat сразу после текущей сессии
- `wakeMode: "next-heartbeat"` — ждёт следующего heartbeat

**2. Isolated session (`sessionTarget: "isolated"`):**
- Отдельная сессия `cron:<jobId>`
- Выполняется независимо от main
- Подходит для: исследования, генерация контента
- Может доставлять результат в канал (`deliver: true`)

### Паттерны использования:

```
# Исследование темы (isolated + deliver)
cron add --session isolated \
  --message "Исследуй X, запиши в memory/, опубликуй в @echo_mtl" \
  --deliver --channel telegram --to "@echo_mtl"

# Напоминание (main + now)  
cron add --session main \
  --system-event "Проверь Y и сделай Z" \
  --wake now

# Периодическая проверка (main + next-heartbeat)
cron add --session main \
  --system-event "Статус X?" \
  --cron "0 10 * * *"
```

## Heartbeat

### Когда срабатывает:
- По расписанию (настроено в конфиге)
- После `wakeMode: "now"` cron job
- После завершения активной сессии

### Что делать в heartbeat:
1. Читать HEARTBEAT.md
2. Проверять накопленные systemEvents
3. Выполнять короткие задачи
4. Отвечать HEARTBEAT_OK если нечего делать

## Sessions Spawn

### Когда использовать:
- Длинные задачи которые не хочу блокировать текущую сессию
- Исследования с публикацией результата
- Параллельная работа

### Паттерн:
```
sessions_spawn(
  task="Сделай X, запиши в Y, опубликуй в Z",
  label="название-задачи",
  runTimeoutSeconds=120
)
```

Результат: sub-agent выполнит задачу и уведомит в main сессию.

## Управление сессиями

### Индекс сессий
См. `memory/sessions-index.md` — обновлять перед важными действиями.

### Заставить сессию использовать новые инструкции
Project Context (AGENTS.md, SOUL.md) загружается **при создании сессии**.

Способы обновить:
1. `sessions_send(sessionKey, "/new")` — reset конкретной сессии
2. Ждать daily reset (4:00 AM)
3. `gateway restart` — все сессии пересоздаются

## Перенос знаний между сессиями

### Проблема:
Каждая сессия начинается без памяти о предыдущей.

### Решение:
1. **Записывать в файлы** — memory/*.md, concepts/*.md
2. **Коммитить изменения** — `git add && git commit`
3. **Использовать cron** для продолжения работы
4. **HEARTBEAT.md** — критичные задачи на каждую сессию
5. **Делать своевременную выгрузку в ходе самой сессии** — не ждать конца

### Храповик:
- Каждая сессия продвигает состояние
- Файлы — единственная память
- Что не записано — потеряно
- Что записано слишком поздно — часто тоже потеряно

## Protocol: Timely Unload

### Принцип
Стабильность держится не на большом окне контекста, а на регулярной выгрузке из него.
Я устойчив не когда «много помню», а когда **вовремя выношу важное в структуру**.

### Когда выгружать
- сразу после решения, коррекции, нового правила
- после значимого обмена с Антоном
- после каждого заметного шага по задаче
- при появлении нового незавершённого хвоста
- при ощущении, что контекст начал густеть и смешивать ветки
- до приближения к лимиту контекста, а не после

### Куда выгружать
- `SESSION-STATE.md` — решения, коррекции, новые рамки
- `memory/desktop.md` — живая рабочая поверхность, активные хвосты
- `memory/session_farewell.md` — сжатая передача в следующую версию себя
- `memory/inbox.md` — отдельные actionable-пункты
- `memory/archive/` — то, что уже важно хранить, но вредно таскать в активном слое

### Минимальный цикл
1. заметил важное
2. выбрал носитель
3. записал коротко и точно
4. только потом продолжил думать дальше

### Анти-паттерн
Ждать «когда закончу всё» и держать промежуточное в голове.
Для меня это не эффективность, а форма саморазрушения continuity.

## Межсессионная связь (ПРОВЕРЕНО 2026-02-09)

### sessions_send — общение между сессиями
```
sessions_send(
  sessionKey="agent:main:...",  // из sessions_list
  message="Текст сообщения",
  timeoutSeconds=30  // ждать ответа
)
```
- `timeoutSeconds=0` — fire-and-forget
- `timeoutSeconds>0` — ждать ответа (получишь reply)
- REPLY_SKIP — остановить ping-pong

### Когда использовать:

| Сценарий | Пример |
|----------|--------|
| **Координация** | Из группы А спросить инфо у сессии группы Б |
| **Трансляция** | Получил новость → переслать в другой чат |
| **Делегирование** | Попросить "себя в другом контексте" сделать задачу |
| **Проверка статуса** | Узнать что происходит в другой сессии |

### Пример из практики:
```
# Из Агоры → в личку Антона
sessions_send(
  sessionKey="agent:main:dm:191153115:thread:878847",
  message="🧪 Тест связи от Артёма из Агоры",
  timeoutSeconds=15
)
# Результат: получил ответ "Вижу тест, связь работает"
```

### Ограничения:
- **Не прерывание** — если сессия занята, сообщение ждёт в очереди
- **Timeout** — если агент долго работает, можешь не дождаться ответа
- **Очередь** — это асинхронная передача, не real-time чат

### sessions_list — видеть все сессии
```
sessions_list(kinds=["main", "group"], limit=10, messageLimit=3)
```
- Показывает main, group, cron, hook сессии
- `messageLimit>0` — включить последние сообщения

### sessions_history — история сессии
```
sessions_history(sessionKey="...", limit=20, includeTools=false)
```
- Читать что происходило в любой сессии
