## Делегирование — самая выгодная трата времени (2026-03-18)
Всё что можно делегировать — делегировать. Даже если настройка дорогая.
Издержки настройки делегации окупаются масштабированием по цепочке.
Улучшать качество делегирования важнее чем делать самому.
**Формула:** Hub оркестрирует → агенты исполняют → Hub не делает то, что может агент.
**Следствие:** когда задача появляется — первый вопрос не "как сделать", а "кому делегировать".

## Telegram

### L-TG001 — Добавить фото к опубликованному посту = editMessageMedia, не удалять

**Контекст:** 2026-03-10, добавлял иллюстрации к постам 603 и 605. Удалил и перепубликовал вместо редактирования.

**Правило:** Если нужно добавить/заменить медиа в уже опубликованном посте — использовать `editMessageMedia`:
```
telegram_raw(callApi, editMessageMedia, {
  chat_id: <id>,
  message_id: <id>,
  media: { type: "photo", media: "<url>", caption: "<текст>" }
})
```
Это сохраняет message_id, историю репостов и не создаёт новый пост.
Удаление + перепубликация = потеря ID и истории.

## Контекст-изоляция

### L014 — НЕ смешивать контексты (MTLA ≠ локальные группы)
Категорически запрещено упоминать локальные группы/регионы в публичных MTL-контекстах. MTLA/Agora — глобальное пространство, без географической привязки.

**Исключение:** легальные упоминания ("российская группа", "Moscow group", "русскоязычное комьюнити").

**Запрещено:** токсичные регионы ("донбасская", "крымская", "сирийская" и т.д.) — геополитически заряженные территории. *(2026-03-07)*

### Публикация фото+текст в Telegram канал (S007, 2026-03-17)
**Проблема:** message tool не поддерживает HTML parse_mode в caption. sendPhoto через raw API не принимает локальные файлы (нужен URL или InputFile). Попытка объединить фото+caption через message tool даёт сырой HTML-код вместо форматированного текста.
**Решение:** Публиковать ДВУМЯ сообщениями: 1) фото через `message(action=send, filePath=...)` 2) текст через `telegram_raw(sendMessage, parse_mode=HTML)`. НЕ пытаться делать caption с HTML.
**Альтернатива:** Загрузить фото на хостинг (Telegraph, fal.ai URL) → sendPhoto через raw API с URL + caption + parse_mode=HTML.
**Никогда не удалять** уже опубликованный пост, чтобы «переделать» — лучше editMessageText/editMessageCaption.

## 2026-03-19 — Редактирование постов Telegram с гиперссылками

**Проблема:** `message tool` с `action=edit` и `telegram_raw.editMessageText` теряют все entities (ссылки) при редактировании.

**Причина:** оба инструмента передают plain text без parse_mode → ссылки стираются.

**Решение:** для редактирования постов с гиперссылками — ТОЛЬКО curl напрямую:
```bash
curl -s -X POST \
  "https://api.telegram.org/bot<TOKEN>/editMessageText" \
  -H "Content-Type: application/json" \
  -d @/tmp/payload.json
```
Где payload.json содержит `parse_mode: "HTML"` и ссылки в формате `<a href="URL">текст</a>`.

**Правило:** перед редактированием поста с ссылками — ВСЕГДА использовать curl + HTML. Никогда не использовать message tool или telegram_raw для редактирования постов с гиперссылками.

## Four-corner experiment (2026-03-20) — провал
- Проблема: введена четырёхуголная архитектура (Hub + P/I/E) для масштабирования. Результат: взрывной рост токенов, спам сессий, потеря контроля над spawned агентами, белый шум в чатах.
- Причины:
  1. Spawned сессии изолированы — Hub не может их прервать или перенаправить.
  2. Нет фильтрации низкоприоритетных heartbeat-сообщений — каждый публикуется в чат.
  3. Сессии создаются пачками (heartbeat + spawns) — огромный расход токенов.
- Вывод: Архитектура не подходит для текущей инфраструктуры. Требуется либо встроенный контроль над сессиями (прерывание, промежуточные статусы), либо отказаться от spawn в пользу внутрипроцессорных модулей.
- Действие: откат. Отключен cron daily-standup, перезапуск gateway для уничтожения corner-сессий. Возврат к Hub-only режиму.
