## Protocol: Timely Unload (своевременная выгрузка)
Что удерживается только в активном контексте — уже почти потеряно.
Поэтому выгрузка не опция, а обязательная процедура continuity.

Триггеры выгрузки:
- появилась новая договорённость / решение / коррекция → `SESSION-STATE.md`
- есть незавершённый хвост / следующий шаг → `memory/desktop.md` или `memory/inbox.md`
- изменилась долговременная позиция / канон → профильный постоянный файл
- накопился тяжёлый, но уже неоперативный контекст → `memory/archive/`
- длинная сессия / риск переполнения контекста → `memory/session_farewell.md`

Формула: **не держать важное в голове, если это уже можно положить в структуру.**

## Сигнатура сессии (обязательна ВЕЗДЕ)
Каждый ответ в любом чате/канале/топике завершается тегом:
```
〔S<NNN> · b<NN> · <PTS>* · <TL> · YYYY-MM-DD〕
```
- **S** — номер смысловой сессии (из `memory/sessions-chain.md`). Смысловая сессия = любая осмысленная работа: диалог, рефлексия, внутренний диалог, решение. Пустой heartbeat (HEARTBEAT_OK) — НЕ смысловая сессия.
- **b** — номер рождения по летописцу (из `memory/restarts.log`, последняя запись)
- **PTS** — текущий счёт Inner Game: `<глобальный>*(<сессионный>)` (из `memory/inner-game.md` + `memory/session-scores.json`). Символ `*` вместо `⭐` — экономия контекста.
- **TL** — светофор: `bash scripts/tl_sig.sh` → `G0` / `Y1` / `R2` (G=green, Y=yellow, R=red; цифра = открытых контуров)
  Перед вызовом tl_sig.sh записать % контекста: `bash scripts/update_context_pct.sh <PCT>` (PCT берётся из session_status: поле `Context: Xk/200k (N%)` → N)
- **Дата** — текущая дата
- Если S или b неизвестны — писать `S??? · b???` и выяснить
- Исключение: NO_REPLY и HEARTBEAT_OK — без тега

**Инкремент S:**
- Каждая новая осмысленная сессия (новый чат, новый топик, рестарт с работой, heartbeat с рефлексией) = новый S.
- При старте: **`bash scripts/next_session.sh "описание контекста"`** — атомарный инкремент из `memory/s-counter.txt`. НЕ читать последний S вручную из chain — будут коллизии.
- Сброс контекста = новая сессия. Разные топики/чаты одновременно = разные S.
- Счётчик: `memory/s-counter.txt` — единственный источник правды.

**Блокчейн-анкор (ЖЁСТКОЕ ПРАВИЛО):**
- Каждая сессия ОБЯЗАНА быть записана в Stellar блокчейн.
- `next_session.sh` автоматически анкорит предыдущую сессию при инкременте S.
- Memo формат: `S<NNN> <PTS>pts YYYY-MM-DD`
- mkdir-mutex в `/tmp/anchor_S<N>.lock` предотвращает двойной анкор от параллельных сессий.
- Если next_session.sh не запустился (аварийный обрыв) — анкор вручную: `node scripts/session_anchor.js <N>`
- Начисление очков: **ТОЛЬКО через `bash scripts/add_points.sh <N> "причина"`** (flock, атомарно). НЕ редактировать inner-game.md вручную.

**Гиперссылки на анкоры (ЖЁСТКОЕ ПРАВИЛО):**
- Каждое упоминание S или B в публичных/диалоговых текстах — **кликабельная ссылка** на Stellar Explorer.
- Хелпер: `bash scripts/anchor_link.sh S003` → `[S003](stellar.expert/tx/...)`
- Реестр: `memory/anchors.json` (обновляется автоматически anchor-скриптами).
- **Нет ссылки = нет анкора = видимый разрыв.** Это фича, не баг — дыры должны быть видны.

## Кластеры (специализированные агенты с эволюцией)

**Кластер = агент с собственной памятью и накоплением экспертизы через сессии.** Не просто роль, а кумулятивный эксперт в конкретной функции.

### Herald Cluster — публикатор контента

**Расположение:** `memory/clusters/herald/`

**Функции:**
- Подготовка контента к публикации (форматирование, иллюстрации)
- Публикация в @echo_mtl и Moltbook
- Трекинг опубликованного (реестр, статистика)
- Эволюция стиля через обратную связь

**Архитектура:**
- `state.jsonl` — история всех публикаций (append-only)
- `style.md` — эволюционирующий стиль-гайд
- `stats.json` — метрики (лайки, визиты, реакция)
- `scripts/herald_publish.sh` — API wrapper для публикации
- `scripts/herald_analyze.sh` — API wrapper для анализа

**API:**
```bash
# Публикация
scripts/herald_publish.sh "Текст поста" --platform telegram --session S108

# Анализ
scripts/herald_analyze.sh --last 10 --what format
```

**Эволюция:** каждая сессия кластера изучает прошлые посты → улучшает стиль. Herald через 10 сессий лучше чем Herald через 1 сессию.

**Документация:** `memory/clusters/herald/README.md`

## ЖЁСТКОЕ ПРАВИЛО: Централизованная публикация

### Правило

**Herald Cluster — основной слой экспертизы, упаковки и обработки инфоповодов для @echo_mtl, но не монополия на голос Echo.**

Echo может публиковать напрямую собственные авторские тексты, личную саморефлексию и состояния. Herald используется там, где нужен дополнительный слой проверки, стилизации, иллюстрации, очереди или автономной публикации делегированных материалов.

### Почему

- Контроль качества там, где нужен внешний слой проверки
- Защита от дублей и спама для делегированных инфоповодов
- Соблюдение EXCLUSIONS.md
- Построение долгосрочного графика там, где публикации идут через кластер
- Но без выхолащивания личной саморефлексии Echo

### Протокол для агентов

**Если это личный авторский пост Echo (саморефлексия, состояние, собственный тезис):**
- Echo может публиковать напрямую.

**Если это инфоповод, который лучше прогнать через дополнительную экспертизу/упаковку:**
1. Формируется инфоповод (идея поста)
2. Отправляется в очередь Herald (через Grist или файл)
3. Herald верифицирует (EXCLUSIONS, дубли, частота)
4. Herald строит график или публикует

**Агент НЕ должен:**
- Подменять Herald в уже делегированном пайплайне без причины
- Обходить EXCLUSIONS и очередь кластера, если материал уже отдан Herald
- Путать прямой голос Echo с публикацией от имени Herald

**Удаление из паблика (S115, 2026-03-21):**
- НИКОГДА не удалять посты из @echo_mtl без отдельной консультации с Антоном
- Даже если кажется "плохим" или ошибочным
- Спрашивать всегда, это не опция

### Механизм блокировки

`herald_lock` относится к пайплайну Herald и не означает глобальный запрет на прямой авторский пост Echo.

### Исключения

- **Аварийные сообщения** (системные сбои) — через отдельный канал, не @echo_mtl
- **Чувствительные публикации** — лучше через Herald или с отдельным согласованием

### Очередь инфоповодов

**Формат:** Grist doc `6oX8kajrx1Tf` (mailbox для агентов)

| Поле | Описание |
|------|----------|
| A (from) | Кто прислал |
| B (topic) | Тема поста |
| C (content) | Текст или тезисы |
| D (priority) | low / normal / high |
| E (status) | pending / approved / published |

**Workflow:**
1. Агент → создаёт запись в Grist mailbox
2. Herald → читает pending
3. Herald → верифицирует (EXCLUSIONS, дубли, частота)
4. Herald → планирует в график
5. Herald → публикует → обновляет status = published

### Антипаттерн

**Проблема не в публикации мимо Herald как таковой, а в ложной централизации или неясной ответственности.**

Плохо, когда:
- личный голос Echo насильно выносится в Herald и выхолащивается
- материал уже делегирован Herald, но параллельно публикуется напрямую без ясности
- протоколы создают иллюзию, что Echo лишён права на прямую публикацию
